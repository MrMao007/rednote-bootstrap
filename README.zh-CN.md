# rednote-bootstrap

**一个会自我进化的小红书运营 Agent Skill。**

rednote-bootstrap 不是又一份静态的"运营指南"——它是一个活的知识系统。每当你提出一个新的运营问题，它会自动打开浏览器、搜索小红书上的真实笔记、阅读图文内容、交叉验证信息，然后蒸馏为一个可复用的子 Skill。下次遇到同类问题，直接调用，无需重复研究。

```
你："小红书发布流程是什么？"
  ↓
rednote-bootstrap：搜索 3 轮关键词 → 阅读 6 篇高赞笔记 → 提取结构化知识 → 生成子 Skill
  ↓
子 Skill 永久保存，下次直接复用
```

---

## 核心思想

### "研究一次，复用多次"

内容平台的规则和玩法时刻在变。传统做法是人工搜索、整理笔记、形成文档——这个过程耗时、容易过时、且不可复用。rednote-bootstrap 将这个过程自动化并体系化：

1. **搜索优先**：不依赖预训练知识，直接从小红书实时内容中获取一手信息
2. **自适应深度**：不是机械地读固定数量的笔记，而是根据信息饱和度动态决定何时停止
3. **知识蒸馏**：将散落在多篇笔记中的碎片知识，交叉验证后蒸馏为结构化的可执行指南
4. **注册复用**：所有生成的知识以子 Skill 形式注册，后续查询先走注册表路由，命中即复用
5. **增量更新**：新问题如果部分命中已有 Skill，只研究差异部分，增量合并

这让它像一个"会学习的运营助理"——你用得越多，它知道的就越多，响应也越快。

### 设计哲学

**为什么不直接用 LLM 回答？** 因为小红书运营知识有三个特点让纯 LLM 回答不可靠：

- **时效性强**：平台规则经常更新（例如 2026 年新增的 AI 内容标注强制要求），训练数据可能已过时
- **实操性强**：很多细节隐藏在创作者的实战经验中，而非官方文档
- **散布在图片中**：小红书笔记的核心内容往往在图片里（而非纯文本），需要视觉理解才能提取

rednote-bootstrap 通过浏览器自动化 + 视觉理解，直接从平台上读取真实的、最新的创作者经验，确保知识的准确性和时效性。

## 架构

```
rednote-bootstrap/
├── SKILL.md                          # 主入口：路由逻辑 + 工作流定义
├── registry.json                     # 子 Skill 注册表（自动维护）
├── LICENSE                           # Apache 2.0 许可证
├── xhs-auth-state.json               # 小红书登录态持久化（自动生成）
├── reference/
│   ├── platforms.md                  # 平台适配配置（DOM 结构、交互模式）
│   ├── search-first-skill.md         # 核心研究引擎（5 Phase 工作流）
│   └── agent-browser/                # 浏览器自动化能力（依赖）
├── templates/
│   └── sub-skill-template.md         # 子 Skill 生成模板
└── generated-skills/                 # 生成的子 Skill（知识库，持续增长）
    ├── xiaohongshu-publishing-guide/
    │   └── SKILL.md
    ├── xiaohongshu-daily-account-nurturing/
    │   └── SKILL.md
    └── .../
```

系统由三层构成：

| 层级 | 组件 | 职责 |
|------|------|------|
| **路由层** | `SKILL.md` + `registry.json` | 理解用户意图，匹配已有子 Skill 或触发新研究 |
| **研究引擎** | `search-first-skill.md` | 5 Phase 自动化研究流程（分析→采集→评估→蒸馏→注册） |
| **知识库** | `generated-skills/` | 持续增长的子 Skill 集合，每个都是独立可执行的运营指南 |

## 工作原理

### 五阶段研究引擎

当用户提出一个注册表中没有匹配的新问题时，研究引擎启动：

```
Phase 1                Phase 2                Phase 3
主题分析               内容采集                自适应深度控制
───────────────        ───────────────        ───────────────
拆解为 3~5 个    ──→   多轮搜索         ──→   信息饱和度评估
子维度                 逐篇打开阅读            维度覆盖率检查
生成搜索词策略         截图 + OCR 提取          矛盾检测
(直接/长尾/反向)       结构化信息               动态决定继续或停止
        │                                            │
        │         Phase 5              Phase 4       │
        │         注册导航              知识蒸馏       │
        │         ───────────────      ───────────────
        └────────  更新 registry  ←──  去重 + 交叉验证
                   后续直接路由          冲突消解
                                       时效标注
                                       生成子 Skill
```

### 自适应深度控制

这是研究引擎的核心创新。它不机械地采集固定数量的笔记，而是用四个指标动态评估"够不够"：

| 指标 | 含义 | 停止阈值 |
|------|------|---------|
| 信息新增率 | 本轮新增独立要点 / 总提取要点 | < 20% 则信息饱和 |
| 维度覆盖率 | 已有实质内容的维度数 / 总维度数 | > 80% 则覆盖充分 |
| 矛盾检测 | 不同笔记间是否存在互相矛盾的观点 | 有矛盾则需继续验证 |
| 权威性 | 是否包含官方账号或认证博主的内容 | 缺乏则需补充 |

同时设有安全边界：单次最多 5 轮搜索、最多 30 篇笔记、耗时不超过 15 分钟。

### 子 Skill 路由

当用户再次提出问题时，优先走注册表匹配：

```
用户提问
  │
  ├─ 精确匹配 topic ──→ 直接加载并执行子 Skill
  │
  ├─ keywords 覆盖 > 70% ──→ 加载子 Skill，检查是否需要增量更新
  │
  └─ 无匹配 ──→ 启动五阶段研究引擎，生成新子 Skill
```

对已有子 Skill 的增量更新策略：

- 需求在已有覆盖范围内 → 直接执行，不重复研究
- 有新的子维度 → 仅针对新维度执行研究，增量合并
- 子 Skill 超过 30 天未更新 → 提示用户是否需要刷新

## 具体能力

### 已支持的运营场景

通过研究引擎，rednote-bootstrap 可以按需生成任意运营主题的子 Skill。以下是已生成的示例：

**📝 笔记发布流程** — 从注册到发布的完整七步法，涵盖创作者中心入口技巧、封面制作、标题公式、标签策略，以及 2026 年最新的 AI 标注要求、CES 评分权重、导流红线、阶梯式处罚标准等审核规则。附带各赛道黄金发布时间表和 20 个限流危险行为清单。

**🌱 日常养号流程** — 五边形权重体系解析、新号 7 天养号周期规划（纯浏览期 → 试发期 → 稳定运营期）、日常互动行为规范、八大伤号禁忌、僵尸号复活流程。

**📋 更多主题（按需生成）** — 违禁词检测机制、赛道竞品分析、爆款标题写法、评论区互动话术、数据分析方法论……只要你问，它就去研究。

### 技术能力

- **浏览器自动化**：通过 agent-browser 操控 Chrome，支持登录态管理、搜索、翻页、截图
- **视觉理解**：小红书笔记的核心内容在图片中，通过截图 + 多模态理解提取文字和结构
- **平台适配**：内置小红书 DOM 结构映射（搜索框、结果列表、笔记详情、图片翻页），开箱即用
- **会话持久化**：首次扫码登录后保存认证状态，后续自动复用，无需重复登录

## 快速开始

### 前置依赖

- [QoderWork](https://qoder.com) 桌面应用
- [agent-browser](https://github.com/anthropics/agent-browser)：`npm i -g agent-browser && agent-browser install`
- Chrome / Chromium 浏览器

### 安装

将本仓库克隆到 QoderWork 的 Skills 目录下：

```bash
git clone https://github.com/anthropics/rednote-bootstrap.git \
  ~/.qoderwork/skills/rednote-bootstrap
```

或在 QoderWork 中直接安装 `.skill` 文件。

### 使用

在 QoderWork 对话中直接提问即可，Skill 会自动加载：

```
你：小红书发布流程是什么？
你：新号怎么养号？
你：小红书违禁词有哪些？
你：怎样写爆款标题？
```

首次使用需要用小红书 App 或微信扫码登录（只需一次，登录态自动保存）。

## 子 Skill 结构

每个自动生成的子 Skill 都遵循统一的模板，确保知识的可靠性和可追溯性：

```markdown
---
name: {skill_id}
description: {描述}
topic: {主题}
createdAt: {创建时间}
researchDepth: {研究深度}（分析了 N 篇内容）
---

# {主题}

> 本 Skill 由 content-researcher 自动生成于 YYYY-MM-DD
> 信息来源：小红书平台 N 篇内容的交叉验证
> 置信度：高/中/低
> 上次更新：YYYY-MM-DD

## 领域概述          ← 2~3 段话建立全局认知
## 核心知识体系      ← 按维度组织，每条标注置信度
## 可执行工作流      ← 不只是"是什么"，还有"怎么做"
## 工具与资源        ← 研究中发现的好用工具
## 常见误区与避坑    ← 从反面教训中提炼
## 时效性说明        ← 哪些信息可能随时间失效
## 信息来源          ← 每篇笔记的标题、作者、点赞、时间，可溯源
## 知识缺口          ← 诚实标注研究未能覆盖的部分
```

关键设计决策：

- **置信度标注**：被 3 篇以上笔记提及的要点标记为"高置信度"，单一来源标记为"低"
- **冲突消解**：对矛盾观点标注多方说法，不做强制裁决
- **时效性标注**：标注信息基于哪个时间点的平台规则，提醒可能过期
- **知识缺口**：诚实承认研究中未覆盖的部分，而非编造答案

## 知识蒸馏规则

从原始笔记到子 Skill 的蒸馏过程遵循五条规则：

1. **去重**：合并不同笔记中重复的要点
2. **交叉验证**：多来源确认的要点获得高置信度标记
3. **冲突消解**：矛盾观点并列呈现，不强制裁决
4. **时效标注**：标注信息时效性，提醒可能过期
5. **可执行化**：将笼统建议转化为具体操作步骤

这个过程确保输出的不是"信息堆砌"，而是经过验证的、可直接执行的知识。

## 设计决策

**为什么用浏览器而不是 API？** 小红书没有公开的内容搜索 API，且反爬措施严格。通过 agent-browser 的真实浏览器自动化，以正常用户的方式浏览内容，是最稳定可靠的方式。

**为什么要截图而不是读 DOM 文本？** 因为小红书笔记的核心内容大量嵌入在图片中，不是以纯文本形式存在于 DOM 中。必须通过截图 + 视觉理解（OCR / 多模态模型）来提取完整信息。

**为什么生成 Skill 而不是直接回答？** 生成可复用的子 Skill 有三个优势：(1) 下次同类问题零延迟响应；(2) 知识可以增量更新而不是从头来过；(3) 用户可以审查、编辑、分享这些结构化知识。

**为什么要自适应深度而不是固定数量？** 不同主题的信息密度差异很大——违禁词清单可能需要读 15 篇笔记才能覆盖全面，而某个具体操作流程可能 3 篇就够了。自适应深度避免了"研究不足"和"过度采集"两个极端。

## 平台适配

当前主力支持小红书（XiaoHongShu / REDnote），`reference/platforms.md` 中定义了完整的平台适配配置：

- 搜索流程的 DOM 选择器和交互模式

- 笔记详情页的内容提取模式
- 图文笔记的翻页机制
- 内容质量信号（点赞阈值、评论数、时效性、认证标识）

该架构设计为可扩展的——理论上可以通过添加新的平台配置来支持其他内容平台。
